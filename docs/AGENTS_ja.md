# AGENTS.md 日本語版

## プロジェクト

BeLib は、ブラウザ上で EPUB / PDF ファイルを管理・閲覧するための個人用Webアプリケーションです。

このプロジェクトでは、広範囲な機能を実装することよりも、小さく動作するMVPを優先します。  
依頼されたタスクに必要なものだけを実装し、将来使うかもしれない機能を先回りして追加しないでください。

## コアドメイン

MVPでは、以下の機能に注力します。

- ユーザー認証
- 書籍管理
- 定義済みカテゴリからのカテゴリ選択
- EPUB / PDF ファイルのメタデータ管理
- ロールベースの書籍閲覧権限
- 読書進捗管理

現在のMVP対象のDBモデルは以下です。

- User
- Role
- Book
- Category
- BookFile
- ReadingInfo
- RoleBookPermission

明示的に依頼されない限り、第二段階のテーブルを追加しないでください。  
第二段階の機能例は、タグ、ブックマーク、メモ、レビュー、読書履歴、著者テーブル、共有URL、アクセスログです。

## 正とする情報源

データベース構造の正とする情報源は `schema.prisma` です。

DB関連のコードを変更する場合は、必要に応じて以下を同期してください。

- `schema.prisma`
- Prismaマイグレーション
- データベース定義書
- ER図
- 関連するTypeScript型

古いカラムや重複した権限ロジックを残さないでください。

特に以下を守ってください。

- `users.role` ではなく `users.role_id` を使う
- `books.role_id` は追加しない
- ロールベースの書籍閲覧権限には `role_book_permissions` を使う
- `book_files.book_file_id` ではなく `book_files.book_id` を使う
- `reading_statuses` ではなく `reading_infos` を使う
- `reading_infos.current_position` は EPUB CFI を保存する可能性があるため、nullable な text とする
- `reading_infos.read_status` のデフォルト値は `unread`
- `books.page_turn_direction` のデフォルト値は `ltr`

## 開発方針

### 1機能1テスト

すべての機能実装またはバグ修正に対して、意味のある自動テストを書ける場合は、少なくとも1つの関連するテストを追加または更新してください。

機能と対応するテスト、または検証手順の両方が対応されていない場合、そのタスクは完了とはみなしません。  
意味のある自動テストを追加できない場合は、最終回答で理由を説明し、手動確認手順を提示してください。

例：

- カテゴリAPIを追加する場合は、カテゴリAPIのテストを追加する
- 書籍一覧フィルタリングを追加する場合は、フィルタリングのテストを追加する
- 読書進捗更新ロジックを追加する場合は、読書進捗のテストを追加する
- 権限バグを修正する場合は、権限の回帰テストを追加する

無関係な複数機能を1つのテストにまとめないでください。  
1つの振る舞いを明確に検証する小さなテストを優先してください。

ルールを満たすためだけの意味の薄いテストは作成しないでください。  
ドキュメントのみの変更、フォーマットのみの変更、単純な文言変更は、挙動に影響しない限り自動テストを必須としません。

## 実装ルール

- 変更は小さく、焦点を絞る
- 関係ないコードを書き換えない
- 必要がない限り、ファイル名・モデル名・ルート名・コンポーネント名を変更しない
- タスクで求められていない限り、公開されている挙動を変更しない
- 賢そうなコードより、読みやすいコードを優先する
- 早すぎる抽象化を避ける
- UI変更がタスクでない限り、既存のUI構造を維持する
- 明確な理由なく新しいライブラリを追加しない
- ライブラリを追加する場合は、なぜ必要なのかを説明する
- 明示的に依頼されない限り、認証方式を導入・変更しない
- 新しいテストフレームワークを勝手に導入しない。テストコマンドが存在しない場合は、その旨を説明し、手動確認手順を提示する

## 命名ルール

- Prismaのモデル名は PascalCase の単数形にする
- DBテーブル名は `@@map` を使い、snake_case の複数形にする
- DBカラム名は、必要に応じて `@map` を使い、snake_case にする
- TypeScriptの変数名・関数名は camelCase にする
- Reactコンポーネントは PascalCase にする
- 定数は UPPER_SNAKE_CASE にする

例：

- Prisma model: `RoleBookPermission`
- Database table: `role_book_permissions`
- TypeScript variable: `roleBookPermission`

## データベースルール

- リレーションには外部キーを使う
- 重複してはいけない関係には複合ユニーク制約を使う
- `ReadingInfo` は `userId` と `bookId` の組み合わせで一意にする
- `RoleBookPermission` は `roleId` と `bookId` の組み合わせで一意にする
- 平文パスワードを保存しない。`password_hash` を使う
- すでに定義されている箇所では `deleted_at` による論理削除を使う
- 明示的に依頼されない限り、ユーザー個別の書籍権限を追加しない。MVPではロールベースの権限を使う
- 明示的に依頼されない限り、コミット済みの既存マイグレーションファイルを変更しない
- schema変更が必要な場合は、先にschemaを更新し、その後に検証する

## 読書ステータスの値

`read_status` には、以下の値のみを使ってください。

- `unread`
- `reading`
- `completed`

ドキュメント、フロントエンドのフィルター、バックエンドのバリデーション、テストを更新せずに、新しいステータス値を追加しないでください。

## ページめくり方向の値

`page_turn_direction` には、以下の値のみを使ってください。

- `ltr`: left to right
- `rtl`: right to left

`left`、`right`、`horizontal` などの曖昧な値は使わないでください。

## フロントエンドルール

- コンポーネントは小さく読みやすく保つ
- 隠れたグローバル前提よりも、明示的な props を優先する
- ローディング状態には、利用可能であれば既存の skeleton コンポーネントを使う
- テーブルやリストの挙動は予測しやすく保つ
- デザイン変更がタスクでない限り、レイアウトトークンやデザインルールを変更しない
- 実用的であれば、既存の shadcn/ui コンポーネントを使う
- タスクで明確に必要な場合を除き、グローバル状態を導入しない
- タスクで求められていない限り、ルーティングの挙動を変更しない

## バックエンドルール

- DB書き込み前にリクエスト入力をバリデーションする
- 不正な入力、存在しないレコード、権限エラーには明確なエラーレスポンスを返す
- ルートハンドラーは小さく保つ
- 再利用が本当にある場合のみ、共通ロジックを service / helper 関数に切り出す
- 認証と権限チェックなしに、アップロードされたファイルを公開ディレクトリから直接配信しない
- アップロードされた EPUB / PDF ファイルをリポジトリにコミットしない
- タスクで明示的にテスト用fixtureを求められない限り、開発用ファイルはgit管理外のローカル保存場所を使う

## 権限ルール

管理者ユーザーは、書籍とファイルを管理できます。  
一般ユーザーは、自分のロールで許可された書籍のみ閲覧できます。

書籍アクセスは、タスクで明示的に権限モデルを変更しない限り、`role_book_permissions` を通して確認してください。

フロントエンドだけの権限チェックに依存しないでください。  
保護された操作には、バックエンドでの権限チェックが必須です。

## テストルール

タスク完了前に、利用可能な最も関連性の高いチェックを実行してください。  
変更を検証できる最小のコマンドを優先してください。

一般的なチェックには以下が含まれる場合があります。

- `npm run test`
- `npm run lint`
- `npm run typecheck`
- `npx prisma validate`
- `npx prisma format`

コマンドが存在しない場合は、勝手に作ったことにしないでください。  
`package.json` を確認し、利用可能なスクリプトを使ってください。

変更対象の領域にテストフレームワークが存在しない場合は、勝手に導入しないでください。  
代わりに、自動テストコマンドが存在しないことを説明し、手動確認手順を提示してください。

DB schema を変更した場合は、以下を実行してください。

- `npx prisma format`
- `npx prisma validate`

`prisma format` を実行した場合は、フォーマットによる変更差分も含めてください。

マイグレーションが必要な場合でも、タスクで明示的に求められている場合、または schema 変更がタスクに含まれている場合のみ作成してください。

## ドキュメントルール

挙動、schema、セットアップ手順が変わる場合は、ドキュメントを更新してください。

DB変更の場合は、以下を更新してください。

- データベース定義書
- ER図生成を使っている場合はER図
- 関連する場合は seed data のメモ

可能であれば、`schema.prisma` からドキュメントを生成することを優先してください。

## Git / 変更管理

- 差分は焦点を絞る
- 変更内容を要約する
- 実行したテストとその結果を明記する
- 実行していないテストがある場合は、その理由を明記する
- 失敗しているテストを隠さない
- 秘密情報、`.env` ファイル、ローカルDBダンプ、アップロードされた書籍ファイルをコミットしない
- 期待されるプロジェクト成果物でない限り、生成ファイルを差分に含めない

## 最終回答フォーマット

タスク完了時は、以下の形式で回答してください。

1. 変更内容の要約
2. 実行したテスト・チェック
3. 注意点または次の作業

完了できなかったことがある場合は、明確に記載してください。
